The Utility of Git

Git is the first free Distributed version control system to gain widespread popular visibility. As Linus says, distribution, rather than any particular implementation, is the key interesting idea. So how does a distributed version control differ from a centralized system? Let's turn to wikipedia for this one, here are some key features of distributed systems:

  • There may be many "central" repositories.
  • Code from disparate repositories are merged based on a web of trust, i.e., historical merit or quality of changes.
  • Lieutenants are project members who have the power to dynamically decide which branches to merge.
  • Network is not involved in most operations.
  • A separate set of "sync" operations are available for committing or receiving changes with remote repositories.
So the essence here is a Distributed system means each checkout pulls a full copy of the source repository, for all practical purposes indistinguishable from the original. This behavior provides some features and some implementation challenges, first let's look at what we get:
  • true offline development, because we have our own copy, nearly every feature of the system can be utilized without network connectivity.
  • local commits, branching, and tagging - we can use revision control at every stage of development and only share our changes when ready.
  • fast operations - because the network is not involved, pulling blame annotated source, commit statistics, or historical spelunking all can run on the local system.
  • redundancy - every checkout produces an organic backup of the central repository
  • reduced server load - because the server is involved in a reduced set of operations, it's not working as hard.

Is this simple idea new for revision control: Do more on the client? Let's consider a few representative revision control systems over the course of time: RCS, CVS, and Subversion. In RCS there was no reasonable notion of checking out source, you worked on the server. CVS brought us the ability to check out source, but every revision control operation would hit the network. In subversion a checkout keeps a pristine copy of each file so you can do local diffing. Now git keeps the whole history, so you can do local, well, anything. Framed in this way distribution seems a logical evolution. We've been slowly moving more and more data and functionality to the client, and now we've taken it to its logical end.

While technical characteristics of distribution are interesting, the sociological elements are moreso. Because a checkout gives you a full copy of the source repository, things like "commit priviledges" are no longer meaningful. Everyone may commit to their personal repository. No one is special. If I were to clone a public open source repository, and host it somewhere, now I have a copy that's the same as the original, and if I do a good job, folks can follow me rather than the original author.
